Automatización con GitHub Actions

👤 Oscar de la Cruz
📅 20 Feb 2026
⏱️ 16 min lectura

Introducción

GitHub Actions es la plataforma de CI/CD integrada de GitHub que permite automatizar workflows de desarrollo, testing y deployment. En esta guía completa exploraremos cómo configurar pipelines eficientes para tus proyectos, desde pruebas automatizadas hasta despliegues automáticos.

Con GitHub Actions, puedes construir flujos de trabajo personalizados directamente en tu repositorio. Cada workflow se compone de jobs que ejecutan steps, y puedes activarlos por eventos como pushes, pull requests, o schedule.

Conceptos Fundamentales

Para entender GitHub Actions, es importante conocer sus componentes principales:

Arquitectura Básica

  • Workflow: Proceso automatizado completo que define tu pipeline
  • Job: Conjunto de steps que se ejecutan en un runner
  • Step: Acción individual dentro de un job
  • Runner: Máquina que ejecuta los jobs
  • Action: Script reutilizable para tareas comunes
  • Trigger: Evento que inicia el workflow

Tipos de Runners

  • GitHub-hosted: Linux, Windows, macOS proporcionados por GitHub
  • Self-hosted: Tus propias máquinas con más control
  • Larger runners: Más recursos para proyectos grandes
💡 Tip Profesional
Empieza con runners GitHub-hosted para desarrollo y considera self-hosted cuando necesites entornos específicos o mejor rendimiento.

Estructura de Workflows

Los workflows de GitHub Actions se definen en archivos YAML ubicados en .github/workflows/. Cada workflow puede tener múltiples jobs que se ejecutan en paralelo o secuencia.

Workflow Básico

Ejemplo de Workflow Completo
# .github/workflows/main.yml
name: CI/CD Pipeline

# Trigger - Cuándo se ejecuta
on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]
  schedule:
    - cron: '0 2 * * *'  # Todos los días a las 2 AM

# Variables de entorno
env:
  NODE_VERSION: '18'
  PYTHON_VERSION: '3.11'

# Jobs - Tareas a ejecutar
jobs:
  # Job de testing
  test:
    name: Run Tests
    runs-on: ubuntu-latest
    
    strategy:
      matrix:
        node-version: [16, 18, 20]
    
    steps:
      - name: Checkout code
        uses: actions/checkout@v4
      
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: ${{ matrix.node-version }}
          cache: 'npm'
      
      - name: Install dependencies
        run: npm ci
      
      - name: Run linting
        run: npm run lint
      
      - name: Run tests
        run: npm run test:coverage
      
      - name: Upload coverage
        uses: codecov/codecov-action@v3
        with:
          token: ${{ secrets.CODECOV_TOKEN }}

Workflow Multi-OS

Testing Multiplataforma
# .github/workflows/cross-platform.yml
name: Cross Platform Testing

on: [push, pull_request]

jobs:
  test:
    name: Test on ${{ matrix.os }}
    runs-on: ${{ matrix.os }}
    
    strategy:
      matrix:
        os: [ubuntu-latest, windows-latest, macos-latest]
        python-version: ['3.9', '3.10', '3.11']
        exclude:
          - os: windows-latest
            python-version: '3.9'
    
    steps:
      - name: Checkout
        uses: actions/checkout@v4
      
      - name: Setup Python
        uses: actions/setup-python@v4
        with:
          python-version: ${{ matrix.python-version }}
      
      - name: Install dependencies
        run: |
          python -m pip install --upgrade pip
          pip install -r requirements.txt
      
      - name: Run tests
        run: python -m pytest

Actions Comunes

GitHub Actions proporciona un ecosistema rico de actions predefinidas para tareas comunes. Estas actions te permiten ahorrar tiempo y seguir mejores prácticas.

Checkout

Action de Checkout
- name: Checkout repository
  uses: actions/checkout@v4
  with:
    # Opciones comunes
    fetch-depth: 0           # Clonar todo el historial
    token: ${{ secrets.PAT }}  # Token personalizado
    path: 'subdirectory'     # Clonar en subdirectorio

Setup de Lenguajes

Setup Actions para Diferentes Lenguajes
# Node.js
- name: Setup Node.js
  uses: actions/setup-node@v4
  with:
    node-version: '18'
    cache: 'npm'
    cache-dependency-path: '**/package-lock.json'

# Python
- name: Setup Python
  uses: actions/setup-python@v4
  with:
    python-version: '3.11'
    cache: 'pip'

# Go
- name: Setup Go
  uses: actions/setup-go@v4
  with:
    go-version: '1.21'

# Java
- name: Setup Java
  uses: actions/setup-java@v3
  with:
    distribution: 'temurin'
    java-version: '17'

Cache

Optimización con Cache
- name: Cache dependencies
  uses: actions/cache@v3
  with:
    path: |
      ~/.npm
      ~/.cache/pip
      target/
    key: ${{ runner.os }}-deps-${{ hashFiles('**/package-lock.json') }}
    restore-keys: |
      ${{ runner.os }}-deps-

Testing Automatizado

El testing automatizado es una parte fundamental de CI/CD. GitHub Actions facilita la ejecución de diferentes tipos de pruebas en cada commit.

Testing Multi-Lenguaje

Pipeline de Testing Completo
name: Comprehensive Testing

on: [push, pull_request]

jobs:
  test-node:
    name: Node.js Tests
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '18'
          cache: 'npm'
      
      - run: npm ci
      - run: npm run lint
      - run: npm run test:coverage
      - uses: codecov/codecov-action@v3

  test-python:
    name: Python Tests
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v4
        with:
          python-version: '3.11'
          cache: 'pip'
      
      - run: pip install -r requirements.txt
      - run: flake8 .
      - run: mypy .
      - run: pytest --cov=./ --cov-report=xml --cov-report=html
      - uses: codecov/codecov-action@v3

⭐ Mejores Prácticas de Testing

  • Ejecuta linting antes que las pruebas unitarias
  • Usa matrices para probar múltiples versiones
  • Implementa coverage reporting
  • Cache dependencies para mejorar rendimiento
  • Separa tests unitarios de tests de integración

Build y Deploy

Una vez que las pruebas pasan, el siguiente paso es construir y desplegar tu aplicación. GitHub Actions facilita este proceso con múltiples opciones de despliegue.

Build de Aplicación Web

Pipeline de Build y Deploy
# .github/workflows/deploy.yml
name: Build and Deploy

on:
  push:
    branches: [ main ]

jobs:
  build:
    name: Build Application
    runs-on: ubuntu-latest
    
    steps:
      - name: Checkout
        uses: actions/checkout@v4
      
      - name: Setup Node.js
        uses: actions/setup-node@v4
        with:
          node-version: '18'
          cache: 'npm'
      
      - name: Install dependencies
        run: npm ci
      
      - name: Run tests
        run: npm run test
      
      - name: Build application
        run: npm run build
        env:
          VITE_API_URL: ${{ secrets.API_URL }}
          VITE_APP_VERSION: ${{ github.sha }}
      
      - name: Upload build artifacts
        uses: actions/upload-artifact@v3
        with:
          name: build
          path: dist/
          retention-days: 1

  deploy:
    name: Deploy to Production
    runs-on: ubuntu-latest
    needs: build
    
    steps:
      - name: Download build artifacts
        uses: actions/download-artifact@v3
        with:
          name: build
      
      - name: Deploy to Vercel
        uses: amondnet/vercel-action@v25
        with:
          vercel-token: ${{ secrets.VERCEL_TOKEN }}
          vercel-org-id: ${{ secrets.VERCEL_ORG_ID }}
          vercel-project-id: ${{ secrets.VERCEL_PROJECT_ID }}
          vercel-args: '--prod'
          
      - name: Deploy to Netlify
        uses: nwtgck/actions-netlify@v2
        with:
          publish-dir: './dist'
          production-branch: main
          github-token: ${{ secrets.GITHUB_TOKEN }}
          deploy-message: "Deploy from GitHub Actions"
          enable-pull-request-comment: false
          enable-commit-comment: false
          overwrites-pull-request-comment: false
        env:
          NETLIFY_AUTH_TOKEN: ${{ secrets.NETLIFY_AUTH_TOKEN }}
          NETLIFY_SITE_ID: ${{ secrets.NETLIFY_SITE_ID }}

Deploy Docker

Pipeline Docker
# .github/workflows/docker.yml
name: Build and Push Docker Image

on:
  push:
    branches: [ main ]
    tags: [ 'v*.*.*' ]

jobs:
  docker:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
      
      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3
      
      - name: Login to Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKER_USERNAME }}
          password: ${{ secrets.DOCKER_PASSWORD }}
      
      - name: Extract metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: yourusername/yourapp
          tags: |
            type=ref,event=branch
            type=ref,event=pr
            type=semver,pattern={{version}}
            type=semver,pattern={{major}}.{{minor}}
      
      - name: Build and push Docker image
        uses: docker/build-push-action@v5
        with:
          context: .
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}

Secrets y Seguridad

La seguridad es crucial en CI/CD. GitHub Actions proporciona varias formas de manejar información sensible de manera segura.

Gestión de Secrets

Uso Seguro de Secrets
# Usar secrets en workflows
- name: Deploy with secrets
  run: |
    curl -X POST \
      -H "Authorization: Bearer ${{ secrets.API_TOKEN }}" \
      -H "Content-Type: application/json" \
      -d '{"version": "${{ github.sha }}"}' \
      https://api.example.com/deploy

# Secrets en environment variables
env:
  DATABASE_URL: ${{ secrets.DATABASE_URL }}
  JWT_SECRET: ${{ secrets.JWT_SECRET }}

# Secrets en steps
- name: Configure AWS CLI
  run: |
    aws configure set aws_access_key_id ${{ secrets.AWS_ACCESS_KEY_ID }}
    aws configure set aws_secret_access_key ${{ secrets.AWS_SECRET_ACCESS_KEY }}
    aws configure set default.region ${{ secrets.AWS_REGION }}
🛡️ Mejores Prácticas de Seguridad
  • Nunca incluyas secrets en el código o commits
  • Usa secrets específicos del repositorio cuando sea posible
  • Rota secrets regularmente
  • Usa encrypted secrets para datos sensibles
  • Limita permisos de tokens al mínimo necesario
  • Audita regularmente el uso de secrets

Mejores Prácticas

Para sacar el máximo provecho de GitHub Actions, es importante seguir mejores prácticas que optimicen rendimiento, seguridad y mantenibilidad.

Optimización de Performance

  • Usa cache para dependencies y artifacts
  • Paraleliza jobs cuando sea posible
  • Usa runners específicos para tareas especializadas
  • Optimiza el tamaño de artifacts
  • Usa conditional execution para evitar trabajo innecesario

Mantenibilidad

  • Documenta tus workflows y convenciones
  • Usa templates y actions reutilizables
  • Implementa monitoreo y alerting
  • Usa versiones específicas de actions
  • Prueba workflows localmente con act

Cost Management

  • Monitorea el uso de minutes de runners
  • Usa self-hosted runners para workloads pesadas
  • Optimiza el tiempo de ejecución de jobs
  • Usa caching para reducir descargas
  • Configura timeouts apropiados

⭐ Conclusiones Clave

  • GitHub Actions es una plataforma CI/CD poderosa y flexible
  • Empieza simple y añade complejidad gradualmente
  • Usa cache para mejorar rendimiento
  • Implementa testing automatizado temprano
  • Monitorea costos y optimiza workflows
  • Documenta tus patrones y convenciones
  • Mantén secrets seguros y rotación regular
