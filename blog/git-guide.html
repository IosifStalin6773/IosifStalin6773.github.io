Guía de Git para Desarrolladores

👤 Oscar de la Cruz
📅 5 Feb 2026
⏱️ 16 min lectura

Introducción

Git es el sistema de control de versiones más utilizado en el desarrollo de software moderno. Dominar Git no solo te hace más productivo, sino que es una habilidad fundamental para trabajar en equipo y mantener código de calidad.

Esta guía cubre desde los conceptos básicos hasta técnicas avanzadas que te permitirán manejar cualquier situación en el desarrollo de software. Aprenderás workflows profesionales, comandos esenciales y buenas prácticas que te diferenciarán como desarrollador.

ℹ️ ¿Por qué Git?
Git no solo guarda versiones de tu código, sino que facilita la colaboración, permite experimentar sin miedo, mantiene un historial completo de cambios, y es esencial para el desarrollo moderno.

Conceptos Fundamentales

📚 Terminología Esencial

Repository (Repositorio)

Es el contenedor que almacena todo tu proyecto, incluyendo el código, historial de cambios, configuraciones y metadatos.

Commit

Una instantánea de tu proyecto en un momento específico. Cada commit tiene un ID único (hash SHA-1) y contiene los cambios, autor, fecha y mensaje descriptivo.

Branch (Rama)

Las ramas permiten trabajar en diferentes versiones del proyecto simultáneamente. La rama principal suele llamarse main o master.

💡 Concepto Clave
Git es un sistema distribuido, lo que significa que cada desarrollador tiene una copia completa del historial del proyecto. Esto permite trabajar offline y mantiene redundancia de los datos.

Configuración Inicial

Una configuración adecuada es fundamental para un workflow eficiente. Configura tu entorno Git una vez y olvídate de problemas futuros.

Configurar Identidad

Configuración Básica
# Configurar nombre y email (obligatorio)
git config --global user.name "Tu Nombre"
git config --global user.email "tu@email.com"

# Configurar editor por defecto
git config --global core.editor "code --wait"  # VS Code
git config --global core.editor "nano"         # Nano
git config --global core.editor "vim"          # Vim

# Configurar herramienta de diferencias
git config --global diff.tool "vscode"

# Configurar alias útiles
git config --global alias.st "status"
git config --global alias.co "checkout"
git config --global alias.br "branch"
git config --global alias.cm "commit -m"
git config --global alias.unstage "reset HEAD --"
git config --global alias.last "log -1 HEAD"

# Ver configuración actual
git config --list

Configuración de Seguridad

SSH Keys para GitHub
# Generar clave SSH Ed25519 (recomendado)
ssh-keygen -t ed25519 -C "tu@email.com"

# Iniciar agente SSH
eval "$(ssh-agent -s)"

# Añadir clave privada al agente
ssh-add ~/.ssh/id_ed25519

# Copiar clave pública al portapapeles
cat ~/.ssh/id_ed25519.pub | clip

# Probar conexión
ssh -T git@github.com
🛡️ Seguridad
Las claves SSH son más seguras que los passwords y evitan tener que autenticarte constantemente. Configúralas una vez y úsalas en todos tus repositorios.

Comandos Básicos

Estos son los comandos que usarás diariamente. Dominarlos es fundamental para un workflow eficiente.

Inicializar y Clonar

Crear y Obtener Repositorios
# Inicializar nuevo repositorio
git init mi-proyecto
cd mi-proyecto

# Clonar repositorio existente
git clone https://github.com/user/repo.git
git clone git@github.com:user/repo.git  # SSH

# Clonar en directorio específico
git clone https://github.com/user/repo.git mi-directorio

# Ver estado del repositorio
git status

# Ver historial de commits
git log --oneline --graph --decorate
git log --author="Tu Nombre" --since="1 month ago"

Working Directory vs Staging Area

⚠️ Concepto Clave
Git tiene tres áreas principales: Working Directory (tus archivos), Staging Area (archivos listos para commit), y Repository (historial de commits).
Flujo de Trabajo Básico
# Ver cambios en working directory
git status

# Añadir archivos específicos al staging area
git add archivo.js
git add *.js
git add src/

# Añadir todos los cambios
git add .
git add -A  # Incluye archivos eliminados

# Hacer commit
git commit -m "Mensaje descriptivo del cambio"

# Ver qué cambiará en el próximo commit
git diff --staged

# Ver diferencias entre working y staging
git diff

# Ver diferencias entre commits
git diff HEAD~1 HEAD

Ver y Navegar el Historial

Inspección del Historial
# Ver log con formato personalizado
git log --pretty=format:"%h - %an, %ar : %s" --graph

# Ver cambios específicos de un archivo
git log -p archivo.js

# Buscar commits por mensaje
git log --grep="fix bug"

# Ver commits que modificaron un archivo específico
git log --follow archivo.js

# Ver quién modificó cada línea
git blame archivo.js

# Ver estadísticas de commits
git shortlog -sn

# Ver gráfico de ramas
git log --oneline --graph --all --decorate

Branching

Las ramas son la característica más poderosa de Git. Te permiten trabajar en múltiples versiones simultáneamente sin afectar el código principal.

Crear y Gestionar Ramas

Operaciones con Ramas
# Listar ramas
git branch
git branch -r  # Remotas
git branch -a  # Todas

# Crear nueva rama
git branch feature-nueva-funcionalidad
git checkout -b hotfix-critico  # Crear y cambiar

# Cambiar entre ramas
git checkout main
git switch feature-nueva-funcionalidad

# Renombrar rama actual
git branch -m nuevo-nombre

# Eliminar rama
git branch -d feature-completada  # Solo si está mergeada
git branch -D feature-obsolete    # Forzar eliminación

# Ver diferencias entre ramas
git diff main feature-branch

Workflow de Feature Branch

🌿 Flujo de Trabajo Recomendado
  1. Crear rama para nueva funcionalidad
  2. Trabajar en la rama con commits frecuentes
  3. Hacer merge a main cuando esté completa
  4. Eliminar rama de feature
  5. Repetir para cada nueva funcionalidad

Stashing

Guardar Trabajo Temporal
# Guardar cambios actuales temporalmente
git stash
git stash push -m "Trabajo en progreso"

# Ver stashes disponibles
git stash list

# Aplicar último stash
git stash pop

# Aplicar stash específico y mantenerlo
git stash apply stash@{1}

# Eliminar stash
git stash drop stash@{0}

# Crear rama desde stash
git stash branch feature-from-stash stash@{1}

Merge & Conflicts

Los merges son fundamentales para integrar trabajo de diferentes ramas. Los conflictos son inevitables, pero manejables con las herramientas adecuadas.

Tipos de Merge

Estrategias de Merge
# Fast-forward merge (solo si es posible)
git merge feature-branch --ff-only

# Merge con commit de merge (siempre crea commit)
git merge feature-branch --no-ff

# Squash merge (combina todos los commits en uno)
git merge feature-branch --squash

# Abortar merge en conflicto
git merge --abort

# Continuar merge después de resolver conflictos
git merge --continue

Resolver Conflictos

❌ Conflictos
Cuando Git no puede resolver automáticamente, te mostrará los conflictos. Debes editar los archivos manualmente, elegir qué versión mantener, y luego hacer commit de la resolución.
Marcadores de Conflicto
# Estructura de un conflicto
<<<<<<< HEAD
código en tu rama actual
=======
código en la rama que haces merge
>>>>>>> feature-branch

# Resolver conflicto:
# 1. Editar archivo, dejar solo el código deseado
# 2. Eliminar marcadores <<<<<<<, ======, >>>>>>>
# 3. git add archivo-resuelto
# 4. git commit

Herramientas para Conflictos

Comandos Útiles
# Ver archivos con conflictos
git diff --name-only --diff-filter=U

# Ver conflictos específicos
git diff archivo-en-conflicto

# Elegir una versión automáticamente
git checkout --ours archivo.js      # Mantener tu versión
git checkout --theirs archivo.js     # Mantener versión de la otra rama

# Usar herramienta de merge gráfica
git mergetool

# Configurar herramienta de merge
git config merge.tool vscode
git config mergetool.vscode.cmd 'code --wait $MERGED'

Remote Repositories

Los remotos te permiten colaborar con otros desarrolladores y mantener copias de seguridad de tu trabajo.

Gestionar Remotos

Operaciones con Remotos
# Ver remotos configurados
git remote -v

# Agregar nuevo remoto
git remote add origin https://github.com/user/repo.git
git remote add upstream https://github.com/original/repo.git

# Cambiar URL de remoto
git remote set-url origin git@github.com:user/repo.git

# Eliminar remoto
git remote remove origin

# Ver información de remoto
git remote show origin

# Renombrar remoto
git remote rename origin upstream

Sincronización

Push y Pull
# Enviar cambios al remoto
git push origin main
git push -u origin feature-branch  # Configurar upstream

# Enviar todas las ramas
git push --all origin

# Enviar con tags
git push --tags

# Obtener cambios del remoto
git pull origin main
git fetch origin  # Solo descargar, no hacer merge

# Sincronizar con upstream (forks)
git fetch upstream
git merge upstream/main

# Ver commits que no están en el remoto
git log origin/main..HEAD
git log HEAD..origin/main

Workflows Colaborativos

👥 Pull Requests
  1. Hacer fork del repositorio original
  2. Clonar tu fork
  3. Crear rama para tu contribución
  4. Hacer commits y push a tu fork
  5. Crear pull request en el repositorio original
  6. Responder feedback y hacer cambios si es necesario
  7. Esperar merge y celebrar

Workflow Avanzado

Dominar estas técnicas te permitirá manejar situaciones complejas y optimizar tu flujo de trabajo.

Rebase vs Merge

⚠️ Cuándo usar cada uno
Merge: Preserva historial completo, ideal para ramas principales.
Rebase: Historial lineal, ideal para feature branches antes de PR.
Rebase Interactivo
# Rebase interactivo de últimos 3 commits
git rebase -i HEAD~3

# Comandos disponibles:
# p, pick = usar commit
# r, reword = editar mensaje
# e, edit = modificar commit
# s, squash = combinar con anterior
# f, fixup = combinar descartando mensaje
# d, drop = eliminar commit

# Rebase de rama sobre main
git checkout feature-branch
git rebase main

# Abortar rebase
git rebase --abort

# Continuar rebase después de conflictos
git rebase --continue

Cherry Pick

Seleccionar Commits Específicos
# Aplicar commit específico a rama actual
git cherry-pick 

# Aplicar múltiples commits
git cherry-pick ..

# Cherry pick sin crear commit
git cherry-pick --no-commit 

# Editar mensaje del cherry pick
git cherry-pick --edit 

# Abortar cherry pick
git cherry-pick --abort

Reset y Revert

Deshacer Cambios
# Reset (modifica historial)
git reset --soft HEAD~1     # Mantiene cambios en staging
git reset --mixed HEAD~1    # Mantiene cambios en working directory
git reset --hard HEAD~1     # Elimina todos los cambios

# Reset a commit específico
git reset 

# Revert (crea nuevo commit que deshace cambios)
git revert HEAD
git revert 

# Reset a archivo específico
git reset HEAD -- archivo.js

# Limpiar working directory completamente
git clean -fd

Reflog - El Viajero del Tiempo

🕰️ Recuperación de Datos

El reflog registra todos los movimientos en Git. Es tu red de seguridad cuando creías haber perdido algo para siempre.

Usar Reflog
# Ver reflog
git reflog

# Recuperar commit perdido
git reset --hard HEAD@{2}

# Ver reflog de una rama específica
git reflog feature-branch

# Encontrar commit perdido
git log --walk-reflogs --oneline

Mejores Prácticas

⭐ Convenciones de Commits

  • Formato: tipo(scope): descripción concisa
  • Tipos: feat, fix, docs, style, refactor, test, chore
  • Ejemplo: feat(auth): agregar login con OAuth
  • Presente: "agrega" en lugar de "agregué"
  • Limite: 50 caracteres en la primera línea

Workflow Recomendado

Flujo Profesional
# 1. Mantener main siempre estable
git checkout main
git pull origin main

# 2. Crear rama para cada tarea
git checkout -b feature/nueva-funcionalidad

# 3. Commits atómicos y descriptivos
git add .
git commit -m "feat: implementar base de la funcionalidad"

# 4. Mantener rama actualizada con main
git fetch origin
git rebase origin/main

# 5. Push y crear pull request
git push -u origin feature/nueva-funcionalidad

# 6. Después del merge, limpiar
git checkout main
git pull origin main
git branch -d feature/nueva-funcionalidad
git push origin --delete feature/nueva-funcionalidad

Consejos Profesionales

💡 Tips Expertos
  • Usa .gitignore para excluir archivos innecesarios
  • Haz commits pequeños y frecuentes
  • Nunca haces push directamente a main
  • Usa branches descriptivos
  • Revisa tus cambios antes de hacer push
  • Mantén tu repositorio limpio con garbage collection regular
  • Elige el workflow adecuado para tu equipo y síguelo consistentemente
  • Usa herramientas visuales cuando sea necesario, pero domina los comandos
  • Practica regularmente para mantener fluidez con los comandos básicos
  • Documenta tus convenciones de commits y workflows
  • Siempre prueba antes de hacer push a ramas principales

⭐ Conclusiones Clave

  • Git es una habilidad fundamental para todo desarrollador moderno
  • Los comandos básicos cubren el 80% de las necesidades diarias
  • Los workflows de branching son clave para la colaboración
  • Los conflictos son normales, aprender a resolverlos es esencial
  • El reflog es tu red de seguridad contra errores
  • Las convenciones de commits facilitan la colaboración
  • La práctica constante es la clave para dominar Git
  • Cada equipo tiene su workflow, adáptate pero conoce los fundamentos

Próximos Pasos

Ahora que tienes una base sólida de Git avanzado, considera explorar:

  • Git Hooks: Automatiza tareas con hooks pre-commit, post-merge, etc.
  • Submodules: Gestiona dependencias de otros repositorios
  • Worktrees: Trabaja con múltiples branches simultáneamente
  • Git Attributes: Configura comportamiento específico por archivo
  • Filter Branch: Reescribe historial masivamente (con precaución)
🎓 Recomendación Final
La mejor manera de aprender Git es practicando. Crea un repositorio de prueba, experimenta con estos comandos, y no tengas miedo de cometer errores. Con reflog, siempre puedes recuperar cualquier cambio.
